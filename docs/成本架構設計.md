## 一、設計哲學：為什麼不用「金錢」

### 核心問題

K8s 叢集的「成本」本質是**資源佔用**，而非金錢。現有以金錢估算的方式存在根本性缺陷：

| 問題 | 說明 |
|------|------|
| On-premise 無帳單概念 | 伺服器是固定成本，CPU 使用 50% 或 90% 對公司帳單無差異 |
| 雲端定價極度複雜 | Reserved Instance、Spot、Sustained Use Discount、網路出口費用等無法以 CPU×單價涵蓋 |
| 幣別/定價需人工設定 | 增加無意義的操作負擔，且容易填錯 |
| 估算數字造成誤導 | 用戶可能誤把估算值當作真實帳單 |
| 混合雲場景不一致 | 同一個平台管理 AWS + GCP + 機房，定價完全不同 |

### 正確定位

```
金錢成本   → 由雲廠商帳單系統負責（AWS Cost Explorer、GCP Billing、Azure Cost）
K8s 成本   → 資源佔用效率，屬於「工程治理」範疇
```

**Synapse 的職責**：告訴工程師「資源有沒有被有效利用」，而非「花了多少錢」。

## 三、核心指標定義

### 3.1 資源佔用率（Occupancy）

> 衡量「誰申請了多少叢集容量」，與實際使用無關。

```
CPU 佔用率    = sum(pod CPU requests, namespace) / cluster allocatable CPU
記憶體佔用率  = sum(pod Memory requests, namespace) / cluster allocatable Memory
```

**意義**：即使 Pod 空轉，只要 Request 存在，就佔用了調度器視角的容量。

### 3.2 資源效率（Efficiency）

> 衡量「申請的資源有多少被真正使用」。

```
CPU 效率    = avg(container CPU usage) / sum(container CPU requests)
記憶體效率  = avg(container Memory usage) / sum(container Memory requests)
```

**閾值參考**：
- 效率 < 20%：嚴重過度申請（紅色警示）
- 20%–50%：中度過度申請（橙色提示）
- 50%–80%：合理範圍
- > 80%：可能需要擴容

### 3.3 浪費分數（Waste Score）

```
浪費分數 = (1 - CPU效率) × CPU佔用權重 + (1 - 記憶體效率) × 記憶體佔用權重
```

用於排序，識別最值得優化的 Workload。

### 3.4 容量剩餘（Headroom）

```
CPU Headroom    = (allocatable CPU - sum CPU requests) / allocatable CPU
Memory Headroom = (allocatable Memory - sum Memory requests) / allocatable Memory
```

**注意**：Headroom 基於 requests，不是實際 usage。這是調度器實際使用的容量判斷依據。

## 四、資料來源

### 4.1 Kubernetes API（不需要 Prometheus）

| 資料 | API |
|------|-----|
| 叢集可分配資源 | `GET /api/v1/nodes`，讀取 `status.allocatable` |
| Pod CPU/Memory requests | `GET /api/v1/pods`，讀取 `spec.containers[].resources.requests` |
| Pod CPU/Memory limits | 同上，讀取 `spec.containers[].resources.limits` |
| Namespace 列表 | `GET /api/v1/namespaces` |

**優點**：不依賴 Prometheus，基礎功能在任何叢集均可用。

### 4.2 Prometheus（需要叢集已設定監控）

| 指標 | PromQL |
|------|--------|
| Container CPU 實際用量 | `rate(container_cpu_usage_seconds_total[5m])` |
| Container 記憶體實際用量 | `container_memory_working_set_bytes` |
| P95 CPU 用量（7天） | `quantile_over_time(0.95, rate(container_cpu_usage_seconds_total[5m])[7d:5m])` |
| Pod 重啟次數 | `kube_pod_container_status_restarts_total` |

**降級策略**：無 Prometheus 時，僅顯示 Request-based 佔用分析；效率相關指標顯示「需要 Prometheus」提示。

## 七、資料模型（後端）

### 資源快照表（resource_snapshots）

```sql
CREATE TABLE resource_snapshots (
  id              BIGSERIAL PRIMARY KEY,
  cluster_id      INTEGER NOT NULL REFERENCES clusters(id),
  collected_at    TIMESTAMP NOT NULL,
  -- 叢集整體
  allocatable_cpu    FLOAT NOT NULL,   -- 核
  allocatable_memory FLOAT NOT NULL,   -- GiB
  requested_cpu      FLOAT NOT NULL,
  requested_memory   FLOAT NOT NULL,
  -- 有 Prometheus 才有以下欄位
  usage_cpu          FLOAT,
  usage_memory       FLOAT,
  node_count         INTEGER,
  pod_count          INTEGER,
  created_at      TIMESTAMP DEFAULT now()
);

CREATE INDEX idx_resource_snapshots_cluster_time
  ON resource_snapshots(cluster_id, collected_at DESC);
```

### Namespace 快照表（namespace_snapshots）

```sql
CREATE TABLE namespace_snapshots (
  id              BIGSERIAL PRIMARY KEY,
  cluster_id      INTEGER NOT NULL,
  namespace       TEXT NOT NULL,
  collected_at    TIMESTAMP NOT NULL,
  cpu_request     FLOAT NOT NULL,
  memory_request  FLOAT NOT NULL,
  cpu_usage       FLOAT,              -- nullable，需要 Prometheus
  memory_usage    FLOAT,              -- nullable
  cpu_p95         FLOAT,              -- nullable，7天 P95
  memory_p95      FLOAT,              -- nullable
  pod_count       INTEGER NOT NULL,
  created_at      TIMESTAMP DEFAULT now()
);
```

```sql
CREATE TABLE rightsizing_recommendations (
```

## 八、資料採集排程

### 採集頻率

| 資料類型 | 頻率 | 說明 |
|---------|------|------|
| 叢集資源快照 | 每小時 | 從 K8s API 讀取，低負擔 |
| Namespace 快照 | 每小時 | 含 Prometheus 效率指標（如有） |
| Right-sizing 分析 | 每日 02:00 | 需要 7 天歷史資料，計算量較大 |
| 容量趨勢 | 從快照彙總 | 月底彙總或即時計算 |

### 採集邏輯（偽代碼）

```go
func CollectResourceSnapshot(cluster Cluster) error {
    // Step 1: 從 K8s API 取 allocatable（必定成功）
    nodes, _ := k8sClient.CoreV1().Nodes().List(ctx, metav1.ListOptions{})
    allocatable := sumAllocatable(nodes)

    // Step 2: 取 Pod requests（必定成功）
    pods, _ := k8sClient.CoreV1().Pods().List(ctx, metav1.ListOptions{})
    requested := sumRequests(pods)

    // Step 3: 嘗試從 Prometheus 取 usage（可選，失敗不影響）
    var usage *ResourceUsage
    if cluster.PrometheusURL != "" {
        usage, _ = queryPrometheusUsage(cluster.PrometheusURL)
    }

    // Step 4: 寫入快照
    return db.Save(&ResourceSnapshot{
        ClusterID:   cluster.ID,
        CollectedAt: time.Now(),
        Allocatable: allocatable,
        Requested:   requested,
        Usage:       usage, // 可能為 nil
    })
}
```

## 九、Right-sizing 建議演算法

```
推薦 CPU = max(P95 CPU usage × 1.2, min_cpu)
推薦 Memory = max(P95 Memory usage × 1.25, min_memory)

信心度：
  high   → 有 ≥ 14 天完整 Prometheus 資料
  medium → 有 7-13 天資料
  low    → 資料不足 7 天，建議僅供參考
```

**安全邊際說明**：
- CPU 加 20% 緩衝（CPU 可壓縮，超過 limit 不會被殺）
- Memory 加 25% 緩衝（Memory 不可壓縮，超過 limit 會 OOMKill）

## 十、象限圖設計（效率 vs 佔用）

用於跨 Namespace 比較，快速識別問題區域：

```
記憶體效率
    高 |  閒置浪費     |   健康       |
       |  (低佔 高效率) |  (高佔 高效率)|
       |──────────────────────────────
    低 |  雙重問題     |   過度申請   |
       |  (低佔 低效率) |  (高佔 低效率)|
         低             高    CPU效率
```

- 泡泡大小 = 該 Namespace 佔叢集 CPU request 的比例
- 顏色 = 廢棄分數（綠→紅）
